v1.5-a  
相对于v1.4-alpha的更新内容如下：  
本次更新主要围绕枚举策略分层、配置码格式升级、优化器热路径重构，以及GUI / CLI / 解析链路的一致性补全进行增强。  
重点引入了“完全枚举（纯枚举）”模式，将原有“枚举模式”的语义从单一枚举调整为“大枚举+贪心补充”，同时新增独立的纯枚举入口；配置码系统则从Z7升级为Z8，补充对full_enumeration_mode的持久化支持，并保持对历史配置码的兼容导入。  
除此之外，还修复了非枚举模式下部分场景贪心路径未实际参与最终求解的问题，将枚举候选规模从固定硬编码改为按组合件数与后端类型动态估算，并对C++ / CUDA / OpenCL侧的公共逻辑进行了统一收口，使4件、5件及更高件数组合下的行为更一致、更可控、更易维护。  

[新增]  
1.新增“完全枚举”模式  
GUI高级选项中新增“完全枚举 / 纯枚举”开关  
命令行新增 `--full-enumeration-mode` / `-fenum` 参数  
启用后将只执行纯枚举路径，不再追加贪心+局部搜索补充结果  
适合在希望严格观察纯枚举结果、排查策略差异或进行高强度精确搜索时使用  

2.新增Z8配置码格式  
新版导出默认使用Z8格式  
相较于Z7，presence mask由12位扩展为16位  
用于保存更多配置状态，并为后续字段扩展预留空间  
本次新增的 `full_enumeration_mode` 也已纳入配置码持久化范围  
便于纯枚举方案与普通方案分别保存、分享与复现  

3.新增“枚举+贪心”与“纯枚举”的明确区分  
原先“枚举模式”的语义较为笼统  
现已在GUI日志、CLI参数和优化器入口中明确拆分为两条路径：  
- 枚举模式：大枚举 + 贪心补充，合并后取最优  
- 完全枚举：纯枚举，不追加贪心  
使不同策略的结果来源更加清晰，便于用户理解与比较  

[优化]  
1.优化非枚举模式下的整体求解策略  
非枚举模式不再仅依赖单一路径  
现改为“辅助枚举 + 贪心/局部搜索”协同求解  
并根据组合规模判断采用并行或顺序执行  
降低在不同模组数量、不同件数组合下策略表现波动过大的问题  

2.优化枚举候选规模的控制方式  
旧版枚举预筛选上限偏向固定阈值  
现已改为根据 `combo_size` 与 `compute_mode` 动态估算可承受的枚举规模  
通过组合数预算 `C(N, k)` 反推候选上限  
使4件、5件乃至更高件数时的枚举规模更符合实际算力预算  
避免固定阈值在低件数过保守、高件数又不够精确的情况  

3.优化枚举模式下的预筛选逻辑  
新增面向枚举场景的 `_prefilter_for_enumeration()`  
不再仅按总属性值粗略截断  
而是采用“总分top保底 + 各目标属性top补充 + 合并去重后再截断”的方式构造候选集  
降低关键属性强模组在预筛阶段被过早丢弃的概率  
使大枚举结果更贴近用户真实筛选意图  

4.优化贪心+局部搜索热路径的评分实现  
C++侧为贪心构造与局部搜索新增位图版属性判定路径  
使用 `AttrBitmap` 替代热路径中的 `unordered_set<int>` 查询  
减少高频评分时的哈希查找开销  
在不改变结果口径的前提下，降低大量候选评估时的常数成本  

5.优化GPU / OpenCL后端公共数据组织方式  
新增 `FlattenModulesForGpu()` 与 `UnpackGpuSolutions()` 公共辅助逻辑  
将CUDA / OpenCL两侧原本分散的模组拍平、索引解包、精确分数校验流程统一收口  
减少多后端重复实现  
降低后续继续扩展 `combo_size 1~10`、约束条件或返回格式时出现口径漂移的风险  

6.优化CUDA资源清理流程  
CUDA枚举路径中的显存释放逻辑改为统一 `free_all()` 收口  
正常返回与异常清理共用同一套回收逻辑  
减少重复代码  
降低后续维护时出现“正常路径释放了、异常路径忘了释放”之类问题的概率  

7.优化GUI中的细节性能与展示逻辑  
将特殊属性集合改为模块级常量缓存  
避免在结果渲染、高亮判断等高频UI流程中重复构造 `set(SPECIAL_ATTRS)`  
同时高级选项中的枚举模式文案也改得更明确  
使界面表达更贴近当前真实策略语义  

8.优化模组解析与日志输出开销  
`module_parser.py` 中大量调试字符串构造现已受 DEBUG 级别保护  
仅在真正开启详细日志时才生成详细模组摘要与词条输出  
减少常规运行时无意义的字符串拼接与日志负担  

9.优化双语文本工具的组织方式  
原先多个模块中分别维护 `_t` / `_tr` 一类局部双语函数  
现已统一收敛为 `module_types.py` 中的 `tr()` 公共函数  
减少重复定义  
也便于后续继续补充GUI、CLI与解析层的中英文一致性  

[修复]  
1.修复非枚举模式下某些场景贪心路径未实际参与求解的问题（重要）  
旧版在部分执行分支中  
当候选数量未超过特定阈值时，只执行了枚举路径  
`greedy_solutions` 实际保持为空  
导致“非枚举模式”并没有稳定体现出“贪心+局部搜索 + 枚举补充”的设计目标  
现已重构执行流程  
确保非枚举模式下两条路径都能按预期参与并合并结果  

2.修复高级选项中两种枚举语义可能混淆的问题  
旧版只有单一“枚举模式”开关  
用户难以区分到底是纯枚举，还是带贪心补充的混合策略  
现已新增“完全枚举”开关，并与“枚举+贪心”做互斥处理  
避免同时勾选导致策略语义不明确  

3.修复配置码无法保存“纯枚举”状态的问题  
旧版Z7配置码未包含 `full_enumeration_mode` 字段  
即使后续界面或逻辑扩展出纯枚举概念，也无法完整随配置导出  
现已通过Z8格式补全此项状态  
使导入导出后的策略选择与原始界面保持一致  

4.修复模组优化失败时上层不易感知的问题  
`module_parser.py` 在优化阶段发生异常时  
旧版更偏向日志记录，不利于上层流程准确判断本次求解是否真正失败  
现已在记录错误后继续向上抛出异常  
让GUI / 监控主流程能够更明确地区分“正常完成”和“中途失败”  

5.修复多处双语函数重复维护带来的潜在不一致问题  
旧版 `_t` / `_tr` 分散在多个文件中  
后续新增文案或调整语言逻辑时容易出现某处改了、某处遗漏的问题  
现已统一改为公共 `tr()`  
降低双语口径分叉的风险  

6.修复枚举与结果收尾逻辑分散实现带来的维护隐患  
旧版CPU / CUDA / OpenCL在结果解包、精确评分复核、排序等环节存在多处近似重复逻辑  
随着 `combo_size 1~10`、约束筛选、目标属性boost等能力逐渐扩展  
这类分散实现更容易在后续演进中出现行为偏差  
现已收敛为公共 helper 统一处理  
提高后端之间的一致性与可维护性  

[调整]  
1.调整“枚举模式”的定义  
旧版“枚举模式”更接近“只枚举”  
现调整为“大枚举 + 贪心补充”的混合策略  
如果需要真正的纯枚举  
请使用新增的“完全枚举”模式  

2.调整配置码默认导出前缀  
默认导出前缀由 `Z7:` 调整为 `Z8:`  
用于承载新增状态位与后续扩展空间  
旧版Z7仍可继续导入  

3.调整优化器内部公共逻辑的组织方式  
按类型过滤、属性ID集合构建、最终去重/恢复原始分数/重新排序等收尾逻辑  
已抽取为独立公共函数统一处理  
减少 `optimize_modules()` 与 `enumerate_modules()` 之间的重复代码与分支漂移  

4.调整GPU后端绑定的组织方式  
移除了未实际使用的自动GPU总入口绑定  
改为由Python层按 `compute_mode` 明确选择 CUDA / OpenCL / CPU 路径  
使调用关系更加清晰  
也更便于定位具体后端问题  

5.调整GUI版本显示  
GUI界面版本显示已更新为 `v1.5-a`  
与当前发行阶段保持一致  

[兼容性说明]  
1.本次更新属于“策略分层 + 配置码升级 + 内部热路径重构”版本  
整体GUI使用方式、基础筛选逻辑、核心抓包/离线读取流程仍保持延续性  
旧用户的主要操作习惯无需整体重学  

2.配置码兼容性  
新版默认导出Z8格式  
导入端同时兼容Z8 / 历史Z7 / Z4 / Z1 / 更早的纯Base64(JSON)格式  
旧配置通常无需手动转换即可继续导入  
若旧配置中不存在 `full_enumeration_mode` 字段  
将自动按 `False` 处理  

3.策略结果兼容性  
由于本次调整了：  
- 非枚举模式下辅助枚举与贪心路径的执行方式  
- 枚举候选规模的动态阈值计算  
- 枚举模式下的属性感知预筛选  
- 枚举+贪心 / 纯枚举 的语义拆分  
因此在相同筛选条件下  
部分历史版本的Top-K内容、排序顺序或候选覆盖范围可能与1.4不同  
若出现差异  
通常以1.5的结果为新的策略口径为准  

4.命令行兼容性  
原有 `-enum` / `--enumeration-mode` 仍可继续使用  
但其语义现更偏向“大枚举+贪心”  
若脚本或批处理需要严格纯枚举  
应改用新增的 `-fenum` / `--full-enumeration-mode`
